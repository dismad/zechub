# ZKPとZK-SNARKS

## 要点

- **ZK-SNARKs** = ゼロ知識簡潔非対話型知識証明
- これにより、一方の当事者は情報そのものを明かさずに、**自分が何かを知っていること**を証明できます
- Zcash は ZK-SNARKs を使って、**送信者、受信者、または金額を明かすことなく**、トランザクションが有効であること（正しい金額、未使用の入力）を証明します
- 「Succinct」は、複雑な命題であっても、その証明が非常に小さく、素早く検証できることを意味します
- Orchard プールは、**信頼できるセットアップを必要としない** ZK-SNARK システムである Halo 2 を使用しています

## 証明とは何か？

証明は、あらゆる数学の基礎です。証明とは、あなたが証明しようとしている主張または定理と、その定理が証明されたことを示すために行われる一連の導出のことです。たとえば、三角形の内角の和は 180° である、ということは誰でも独立して確認できます（検証者）。

**証明**

証明者 ---> 主張を行う ---> 検証者が選択する ---> 受理/拒否

（証明者と検証者はどちらもアルゴリズムです）

計算機科学では、効率的に検証可能な証明を指す用語は NP 証明です。これらの短い証明は、多項式時間で検証できます。大まかな考え方は、「ある定理の解が存在し、それが検証者に渡されて確認される」というものです。

### インタラクティブ証明と確率的証明

**インタラクション**: 検証者は証明をただ読むだけでなく、複数回のメッセージのやり取りを通じて証明者と双方向にやり取りします。

**ランダム性**: 検証者から証明者への要求はランダム化されており、証明者はそれぞれに対して正しく応答できなければなりません。

PSPACE - 異なる命題の交替を証明する

### ゼロ知識とは何ですか？

やり取りの後に検証者が計算できることは、事前に証明できたことと同一である。証明者と検証者の間で複数ラウンドにわたる対話を行っても、検証者の計算能力が増すことはない。

**シミュレーション Paradigm**

この実験は暗号技術全般に存在します。そこでは「Real View」と「Simulated View」が提示されます。

リアルビュー：証明者と検証者 (P,V) の相互作用の取りうるすべての履歴

シミュレートされたビュー: 検証者は Prover と Verifier の間で起こりうるすべての相互作用をシミュレートします

</a>

多項式時間判別器は、自分が見ているものが実際のビューなのかシミュレートされたビューなのかを判定しようと試み、両方からサンプルを繰り返し要求します。

2つの見え方は、あらゆる識別アルゴリズム／戦略に対して、たとえ実物またはシミュレーションから多項式個のサンプルを受け取った後であっても、その確率が >1/2 である場合に、「計算量的に識別不可能」であると言われます。

**知識のゼロ知識アーギュメント**

対話型プロトコル (P,V) は、あらゆる確率的多項式時間の検証者に対して（定理が正しい場合に）、実際の視点とシミュレートされた視点を定める確率分布が計算量的に識別不可能となるようなシミュレータ（アルゴリズム）が存在するとき、ゼロ知識である。

インタラクティブ・プロトコルは、検証者が1人だけの場合に有用です。たとえば、ゼロ知識の「納税証明」アプリケーションにおける税務監査官がその一例です。

## SNARKとは何ですか？

**簡潔な非対話型知識の論証**

広義の定義 - ある命題が真であることを簡潔に示す証明。その証明は短く、かつ高速に検証できなければならない。SNARKS では、単一のメッセージが証明者から検証者へ送られる。検証者はその後、受け入れるか拒否するかを選択できる。

例の文: 「SHA256(m)=0 となるメッセージ (m) を私は知っている」

zk-SNARKでは、証明はメッセージ（m）について何も明らかにしません。

**多項式**: 定数（1、2、3 など）、変数（x、y、z など）、および変数の指数（x²、y³ など）を含む項の和。

例: "3x² + 8x + 17"

**算術回路**: 多項式を計算するためのモデル。より一般的には、各グラフノードで算術演算が実行される有向非巡回グラフとして定義できる。回路は加算ゲート、乗算ゲート、およびいくつかの定数ゲートから構成される。ブール回路がワイヤ上でビットを運ぶのと同様に、算術回路は整数を運ぶ。

この例では、証明者は検証者に対して、自分がその算術回路の解を知っていることを納得させたいと考えています。

**コミットメント**: これを行うために、証明者は回路に関連するすべての値（非公開のものと公開のものの両方）をコミットメントに含めます。コミットメントは、出力から元に戻せない関数を使うことで、その入力を隠します。

Sha256は、コミットメントスキームで使用できるハッシュ関数の一例です。

証明者が値にコミットした後、そのコミットメントは検証者に送られます（検証者は、元の値を一切明らかにできないと確信しています）。その後、証明者はグラフの各ノード上のそれぞれの値を知っていることを、検証者に示すことができます。

**フィアット・シャミール変換**

プロトコルを*非対話型*にするために、証明者は暗号学的ハッシュ関数を用いて、検証者に代わってランダム性（隠されたチャレンジに使われるもの）を生成します。これはランダムオラクルとして知られています。証明者はその後、単一のメッセージを検証者に送信でき、検証者はそれが正しいことを確認できます。

汎用回路に利用できる SNARK を構成するには、2 つの要素が必要です。

関数型コミットメント方式: コミッターが短い文字列を使って多項式にコミットでき、その文字列を用いて検証者は、コミットされた多項式の評価値の主張を確認できる。

多項式インタラクティブ・オラクル: 検証者は、多項式コミットメント方式を用いて、証明者（アルゴリズム）に対し、自らが選んださまざまな点で、すべてのコミットメントを開示するよう求め、それらの間で恒等式が真であることを確認する。

**セットアップ**

セットアップ手順は、回路を要約し、公開パラメータを出力することで、検証者を支援します。

**前処理設定の種類**:

回路ごとの Trusted Setup - 各回路につき一度だけ実行されます。これは回路と秘密のランダム性（Common Reference String）に固有のものであり、秘密に保ち、破棄しなければなりません。

この方法における侵害されたセットアップとは、不正な証明者が偽の主張を証明できてしまうことを意味します。

信頼済みかつユニバーサルなセットアップ - 信頼済みセットアップを一度実行するだけで、その後は複数の回路を決定論的に前処理できる。

Transparent Setup（信頼されたセットアップ不要）- 前処理アルゴリズムは、いかなる秘密のランダム性も一切使用しません。


**SNARK証明構成の種類**:

[Groth16](https://www.youtube.com/watch?v=QDplVkyncYQ): 信頼できるセットアップが必要ですが、非常に短い証明であり、迅速に検証できます。

[Sonic](https://www.youtube.com/watch?v=oTRAg6Km1os)/[Marlin](https://www.youtube.com/watch?v=bJDLf8KLdL0)/[Plonk](https://cryptocurrencywiki.org/PLONK): 汎用的なトラステッド・セットアップ.

[DARK](https://www.youtube.com/watch?v=_ZDM7NwSxEY)/[HALO](https://eprint.iacr.org/archive/2019/1021/20200218:011907)/[STARK](https://www.youtube.com/watch?v=wFZ_YIetK1o): 信頼できるセットアップは不要ですが、証明はやや長くなるか、プロバーの実行により長い時間がかかる場合があります。

SNARKは、Zcashのようなブロックチェーンや [Aztec](https://docs.aztec.network) のような zk-Rollup など、複数の検証者が必要な場合に有用です。なぜなら、複数の検証ノードが各証明について何ラウンドにもわたって相互にやり取りする必要がなくなるからです。

## Zcashではzk-SNARKはどのように実装されていますか？

一般に、ゼロ知識証明は、いかなる情報も明かすことなく、プロトコルにおいて誠実な振る舞いを強制するための手段です。

Zcash は、プライベートな取引を可能にするパブリック・ブロックチェーンです。zk-SNARK's は、取引に関するその他の詳細を一切明かすことなく、ネットワークのコンセンサスルールの範囲内でプライベート取引が有効であることを証明するために使用されます。

[動画解説](https://www.youtube.com/watch?v=Kx4cIkCY2EA) - この講義では、Ariel Gabizon が Zcash ノート・コミットメント・ツリー、ブラインド多項式評価、そして準同型的に隠されたチャレンジについて説明し、それらがネットワーク上でどのように実装されているかを解説します。

詳しくは[Halo2ブック](https://zcash.github.io/halo2/index.html)をお読みください。

## その他のゼロ知識アプリケーション

zk-SNARKは、さまざまな異なる用途において複数の利点を提供します。いくつかの例を見てみましょう。

**スケーラビリティ**: これは「計算のアウトソーシング」によって実現されます。L1チェーンがオフチェーンサービスの作業を検証するために、ゼロ知識が厳密に必要というわけではありません。zk-EVM上では、トランザクションが必ずしもプライベートであるとは限りません。

証明ベースのRollup（zk-Rollup）サービスの利点は、数百から数千件のトランザクションのバッチを処理でき、L1 がすべてのトランザクションが正しく処理されたことを示す簡潔な証明を検証できる点にあり、これによってネットワークのトランザクション処理能力を100倍または1000倍に拡張できます。
