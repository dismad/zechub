# FROST 

## Что такое подпись Шнорра?

Подпись Шнорра — это набор алгоритмов: (KeyGen, Sign, Verify).

Подписи Шнорра имеют несколько преимуществ. Одно из ключевых преимуществ заключается в том, что при использовании нескольких ключей для подписания одного сообщения, полученные подписи могут быть объединены в одну подпись. Это может быть использовано для значительного уменьшения размера транзакций с несколькими подписями и других операций с несколькими подписями.

## Что такое FROST?

**Flexible Round-Optimized Schnorr Threshold Signatures** -
*Созданы Chelsea Komlo (Университет Ватерлуо, Zcash Foundation) & Ian Goldberg (Университет Ватерлуо).*

FROST — это протокол пороговой подписи и распределенного генерирования ключей, который обеспечивает минимальное количество раундов коммуникации и безопасность при параллельном выполнении. Протокол FROST является пороговой версией схемы подписи Шнорра.

В отличие от подписей в односторонней среде, пороговые подписи требуют сотрудничества определенного количества подписантов, каждый из которых имеет долю общего приватного ключа. 

[Что такое пороговые подписи? Chelsea Komlo - Zcon3](https://youtu.be/cAfTTfblzoU?t=110)

Следовательно, генерация подписей в пороговой среде требует дополнительных усилий из-за раундов сети между подписантами, что дорого обходится, когда секретные доли хранятся на устройствах с ограниченной пропускной способностью или при координации через ненадежные сети.

Нагрузка на сеть во время операций подписания снижается за счет применения новой техники для защиты от атак подделки, применимой к другим схемам.
 
FROST улучшает пороговые протоколы подписи, позволяя безопасно выполнять неограниченное количество операций подписания параллельно (конкуренция).
 
Его можно использовать как двухраундовый протокол, где подписанты отправляют и получают 2 сообщения в общей сложности, или оптимизировать до однораундового протокола подписания с предварительной стадией. 

FROST достигает своих улучшений по эффективности частично за счет того, что позволяет протоколу завершиться в присутствии недобросовестного участника (который затем идентифицируется и исключается из будущих операций).
 
Доказательства безопасности, демонстрирующие то, что FROST безопасен против атак с выбором сообщения при условии, что задача дискретного логарифма сложна, и вредитель контролирует меньше участников, чем порог, предоставлены [здесь](https://eprint.iacr.org/2020/852.pdf#page=16).

## Как работает FROST?

Протокол FROST содержит два важных компонента:

Первым делом, n участников запускают *протокол распределенного генерирования ключей (DKG)* для создания общего ключа проверки; в конце концов, каждый участник получает приватную долю секретного ключа и публичный ключ проверки.

После этого любые t из n участников могут запустить *протокол пороговой подписи*, чтобы совместно создать действительную подпись Шнорра.

</a>

**Распределенное генерирование ключей (DKG)**

Цель этого этапа — создать долгоживущие доли секретного ключа и общий ключ проверки. Этот этап выполняется n участниками.

FROST строит свою фазу генерирования ключей на основе [Pedersen DKG (GJKR03)](https://blog.gtank.cc/notes-on-threshold-signatures/) , используя как схемы разделения секрета Шамира, так и проверяемые схемы разделения секрета Фельдмана в качестве подпрограмм. Кроме того, каждый участник должен доказать знание своего собственного секрета, отправив другим участникам нулевое знание, которое само по себе является подписью Шнорра. Этот дополнительный шаг защищает от атак с использованием поддельных ключей в условиях, где t ≥ n/2.

В конце фазы DKG генерируется общий ключ проверки vk. Кроме того, каждый участник Pᵢ имеет значение (i, skᵢ), которое является его долгоживущей долей секретного ключа и долей ключа проверки vkᵢ = skᵢ * G. Доля ключа проверки vkᵢ участника Pᵢ используется другими участниками для проверки правильности долей подписи Pᵢ в фазе подписания, тогда как ключ проверки vk используется внешними сторонами для проверки подписей, выданных группой.

**Пороговая подпись**

Этот этап строится на известных техниках, которые используют аддитивное разделение секрета и преобразование долей в непересекающиеся точки эллиптической кривой для генерирования случайного числа для каждой подписи. Этот этап также использует методы привязки, чтобы избежать известных атак на подделку без ограничения конкуренции.

Предварительная обработка: На стадии предварительной обработки каждый участник готовит фиксированное количество пар точек эллиптической кривой (EC) для дальнейшего использования, которая выполняется один раз для нескольких этапов пороговой подписи.

<a href="">
    <img src="https://i.ibb.co/nQD1c3n/preprocess.png" alt="" width="400" height="300"/>
</a>

Раунд подписания 1: Каждый участник Pᵢ начинает с генерирования пары приватных временных чисел (dᵢ, eᵢ) и соответствующих пар точек эллиптической кривой (Dᵢ, Eᵢ), а затем широковещательно передает эту пару точек всем другим участникам. Каждый участник сохраняет полученные пары точек эллиптической кривой для последующего использования. Раунды подписания 2 и 3 — это фактические операции, в которых t из n участников сотрудничают для создания действительной подписи Шнорра.

Раунд подписания 2: Для создания действительной подписи Шнорра любые t участников работают вместе, чтобы выполнить этот раунд. Основная техника за этим раундом — это аддитивное разделение секрета t из t.

Этот шаг предотвращает атаку на подделку, потому что нападающие не могут объединять доли подписи между различными операциями подписания или переставлять наборы подписантов или опубликованные точки для каждого подписанта. 

<a href="">
    <img src="https://i.ibb.co/b5rJbXx/sign.png" alt="" width="400" height="300"/>
</a>

После вычисления вызова c каждый участник может вычислить ответ zᵢ на вызов, используя одноразовые временные числа и долгоживущие доли секрета, которые являются t из n (степень t-1) долями секрета Шамира группы долгоживущего ключа. В конце раунда подписания 2 каждый участник широковещательно передает zᵢ другим участникам.

## Приносит ли это пользу Zcash?

Абсолютно да. Введение FROST в Zcash позволит нескольким сторонам, находящимся в разных географических регионах, управлять расходной властью защищенных ZEC. Преимущество заключается в том, что транзакции, передаваемые с использованием этой схемы подписи, будут неразличимыми от других транзакций на сети, сохраняя сильное сопротивление отслеживанию платежей и ограничивая объем данных блокчейна, доступных для анализа. 

На практике это позволит создать целый ряд новых приложений в сети, начиная от поставщиков эскроу или других неконтролируемых услуг.

FROST также станет важным компонентом безопасного выпуска и управления защищенными активами Zcash (ZSA), обеспечивая более безопасное управление расходной властью в разработочных организациях и хранителях ZEC, таких как биржи, распределяя доверие, а также предоставляя эту возможность пользователям Zcash.

## Использование FROST в более широкой экосистеме

**FROST в [Coinbase](https://github.com/coinbase/kryptology/tree/master/pkg/dkg/frost)**

Чтобы повысить эффективность систем пороговой подписи, разработанных Coinbase, они создали версию FROST. Реализация Coinbase вносит небольшие изменения по сравнению с оригинальным проектом FROST.

Они отказались от использования роли агрегатора подписей. Вместо этого каждый участник является агрегатором подписи. Этот дизайн более безопасен: все участники протокола проверяют вычисления других для достижения более высокого уровня безопасности и снижения риска. Также был удален (одноразовый) этап предварительной обработки, чтобы ускорить реализацию, вместо этого используя третий раунд подписания.

___

**[ROAST](https://eprint.iacr.org/2022/550.pdf) от Blockstream**

Улучшение FROST, специфичное для приложений, предложено для использования на [Blockstream Liquid Sidechain](https://blog.blockstream.com/roast-robust-asynchronous-schnorr-threshold-signatures/) для Bitcoin.

"ROAST — это простой обертка вокруг схем пороговой подписи, таких как FROST. Она гарантирует, что кворум честных подписантов, например, функционариев Liquid, всегда может получить действительную подпись даже в присутствии разрушительных подписантов, когда соединения сети имеют произвольно высокую задержку."

___

**FROST в IETF**

Инженерная группа интернета (IETF), основана в 1986 году, является ведущей организацией по разработке стандартов для интернета. IETF создает добровольные стандарты, которые часто принимаются пользователями интернета, операторами сетей и производителями оборудования, а также помогает формировать траекторию развития интернета.

Версия FROST 11 (двухраундовая версия) была [представлена в IRTF](https://datatracker.ietf.org/doc/draft-irtf-cfrg-frost/11/). 

Это важный шаг для полной оценки FROST как нового стандарта пороговой подписи, предназначенного для использования по всему интернету, в аппаратных устройствах и других сервисах в будущем.
