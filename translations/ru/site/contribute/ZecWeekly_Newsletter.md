# Рассылка ZecWeekly

ZecWeekly — это рассылка, которая выходит каждое воскресное утро. Она включает все новости, произошедшие за неделю в экосистеме Zcash. Новости еженедельно курируются участниками сообщества, и в рассылку добавляются все соответствующие ссылки. Пожалуйста, подпишитесь на рассылку [здесь](https://zechub.substack.com/).

## Участие

Участие в подготовке рассылки работает лучше всего, когда один участник готовит выпуск за соответствующую неделю, следует текущему треду баунти или координации и отправляет pull request после того, как еженедельные ссылки будут готовы. Пожалуйста, не отправляйте будущий выпуск до того, как ZecHub опубликует или подтвердит дату этого выпуска. Слишком ранние pull request часто упускают обновления конца недели, конфликтуют с назначенным куратором или используют неверный дедлайн.

### 1. Подтвердите текущий выпуск

Перед тем как начать писать:

- Проверьте [ZEC Bounties ](https://bounties.zechub.wiki/) на наличие текущего задания по рассылке.
- Дождитесь назначения

### 2. Сделайте форк репозитория

Если вы новичок в GitHub, используйте этот рабочий процесс:

1. Откройте [репозиторий ZecHub](https://github.com/ZecHub/zechub).
2. Нажмите **Fork** и создайте форк в своей учётной записи GitHub.
3. В своём форке создайте новую ветку для выпуска. Понятное имя ветки будет полезным, например `digest-may-30-2026`.
4. Убедитесь, что ваш pull request будет нацелен на `ZecHub/zechub` как базовый репозиторий и на `main` как базовую ветку.

Если вы используете командную строку, тот же рабочий процесс выглядит так:

```bash
git clone https://github.com/YOUR-USERNAME/zechub.git
cd zechub
git checkout -b digest-month-day-year
```

### 3. Создайте файл рассылки

Используйте [шаблон рассылки](https://github.com/ZecHub/zechub/blob/main/newsletter/newslettertemplate.md) в качестве отправной точки. Выпуски рассылки должны находиться в папке [`newsletter`](https://github.com/ZecHub/zechub/tree/main/newsletter).

При создании файла:

- Соблюдайте формат имени файла, указанный в issue или использованный в недавно принятых выпусках.
- Сохраняйте тот же порядок разделов, что и в шаблоне, если только задание не требует другого формата.
- Добавляйте ссылки только за соответствующую неделю.
- Пишите краткое и понятное описание для каждой ссылки, чтобы читатели понимали, почему это важно.
- При необходимости переводите или кратко излагайте неанглоязычные источники на английском.
- Проверьте каждую ссылку перед открытием pull request.

### 4. Собирайте ссылки в правильное время

ZecWeekly обычно охватывает активность экосистемы Zcash за текущую неделю и публикуется ближе к концу недели. Самое безопасное время для этого:

- Начинайте собирать ссылки после публикации текущего выпуска рассылки или задания.
- Держите черновик, пока неделя ещё продолжается.
- Отправляйте pull request ближе к запрошенной дате отправки, после того как вы проверите обновления конца недели.
- Не отправляйте рассылку за будущую неделю до того, как появится задание на эту дату или пока ZecHub не подтвердит, что именно вы должны её подготовить.

Если в issue указано, что нужно отправить материал к определённой дате, следуйте этой дате. Если между этой страницей и текущим issue есть противоречие, следуйте текущему issue.

### 5. Откройте pull request

Когда ваш файл рассылки готов:

1. Зафиксируйте свои изменения в своём форке.
2. Откройте pull request в `ZecHub/zechub` в ветку `main`.
3. Используйте заголовок, соответствующий выпуску, например `Zcash Ecosystem Digest | May 30th`.
4. Укажите issue в тексте pull request, чтобы ревьюеры могли связать работу с заданием.

Пример текста pull request:

```md
Closes #ISSUE_NUMBER

Summary:
- Adds the Zcash Ecosystem Digest for Month Day.
- Uses the newsletter template and the current issue deadline.
- Checks links and descriptions for the requested week.
```

После открытия pull request следите за комментариями ревью. Если ZecHub попросит внести правки, обновите ту же ветку, вместо того чтобы открывать второй pull request для того же выпуска.

### Распространённые ошибки, которых следует избегать

- Открытие pull request до подтверждения даты выпуска или задания.
- Работа над issue, к которому уже привязан pull request.
- Отправка pull request в собственный форк вместо `ZecHub/zechub`.
- Использование неверного имени файла или размещение файла вне папки `newsletter`.
- Копирование старого выпуска без обновления всех дат, ссылок и описаний.
- Добавление ссылок не за ту неделю.
- Наличие неработающих ссылок, дублирующихся ссылок или текста-заполнителя из шаблона.
- Открытие нового pull request после комментариев ревью вместо обновления исходной ветки.

### Финальный чек-лист

Перед запросом ревью убедитесь, что:

- Дата issue или задания совпадает с вашим файлом рассылки.
- Ни один другой открытый pull request уже не охватывает то же issue или выпуск.
- Файл находится в папке `newsletter`.
- Разделы шаблона заполнены полностью.
- Каждая ссылка работает и имеет полезное описание.
- Текст pull request ссылается на правильное issue.
- Вы готовы внести правки, если ревьюеры попросят изменений.
