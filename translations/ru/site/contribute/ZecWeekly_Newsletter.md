# Рассылка ZecWeekly

ZecWeekly — это рассылка, которая выходит каждую пятницу утром. В неё входят все новости, произошедшие за неделю в экосистеме Zcash.

Новости еженедельно курируются участниками сообщества, и в рассылку добавляются все релевантные ссылки.

Пожалуйста, подпишитесь на рассылку [здесь](https://zechub.substack.com/).

## Внести вклад

Вклад в рассылку работает лучше всего, когда один участник готовит выпуск для нужной недели, следует текущей bounty- или координационной ветке и отправляет pull request после того, как еженедельные ссылки готовы. Пожалуйста, не отправляйте будущий выпуск до того, как ZecHub опубликует или подтвердит дату этого выпуска. Слишком ранние pull request часто пропускают обновления конца недели, конфликтуют с назначенным куратором или используют неверный дедлайн.

### 1. Подтвердите текущий выпуск

Прежде чем начать писать:

- Проверьте [issues ZecHub на GitHub](https://github.com/ZecHub/zechub/issues) и [Dework](https://app.dework.xyz/zechub-2424) на наличие текущей задачи по рассылке.
- Используйте дату в заголовке issue или описании задачи как источник истины.
- Откройте issue и проверьте, не оставил ли уже комментарий другой участник, не назначен ли он на задачу и не открыл ли связанный pull request.
- Перед началом найдите открытые pull request по номеру issue и дате выпуска. Например, выполните поиск `is:pr is:open "May 30th" repo:ZecHub/zechub`.
- Если задача неясна, уточните в issue, в Discord ZecHub или напишите [ZecHub в Twitter](https://twitter.com/ZecHub), прежде чем готовить полный выпуск.

![Открытые issues GitHub, отфильтрованные по текущим задачам рассылки ZecWeekly](assets/zecweekly-current-task-search.png)

### 2. Сделайте форк репозитория

Если вы новичок в GitHub, используйте такой рабочий процесс:

1. Откройте [репозиторий ZecHub](https://github.com/ZecHub/zechub).
2. Нажмите **Fork** и создайте форк в своём аккаунте GitHub.
3. В своём форке создайте новую ветку для выпуска. Понятное имя ветки будет полезно, например `digest-may-30-2026`.
4. Убедитесь, что ваш pull request будет направлен в `ZecHub/zechub` как базовый репозиторий и в `main` как базовую ветку.

Если вы используете командную строку, тот же рабочий процесс выглядит так:

```bash
git clone https://github.com/YOUR-USERNAME/zechub.git
cd zechub
git checkout -b digest-month-day-year
```

### 3. Создайте файл рассылки

Используйте [шаблон рассылки](https://github.com/ZecHub/zechub/blob/main/newsletter/newslettertemplate.md) как отправную точку. Выпуски рассылки должны находиться в папке [`newsletter`](https://github.com/ZecHub/zechub/tree/main/newsletter).

При создании файла:

- Соблюдайте формат имени файла, указанный в issue или использованный в недавно принятых выпусках.
- Сохраняйте тот же порядок разделов, что и в шаблоне, если только задача не требует другого формата.
- Добавляйте ссылки только за соответствующую неделю.
- Пишите короткое и понятное описание для каждой ссылки, чтобы читатели понимали, почему она важна.
- При необходимости переводите или кратко пересказывайте неанглоязычные источники на английском.
- Проверьте каждую ссылку перед открытием pull request.

### 4. Собирайте ссылки в правильное время

Обычно ZecWeekly охватывает активность экосистемы Zcash за текущую неделю и публикуется ближе к концу недели. Самое безопасное время такое:

- Начинайте собирать ссылки после того, как будет опубликован текущий issue или задача рассылки.
- Держите черновик, пока неделя ещё продолжается.
- Отправляйте pull request ближе к запрошенной дате отправки, после проверки обновлений конца недели.
- Не отправляйте рассылку за будущую неделю до того, как появится задача на эту дату или пока ZecHub не подтвердит, что именно вам следует её подготовить.

Если в issue указано, что нужно отправить до определённой даты, следуйте этой дате. Если между этой страницей и текущим issue есть расхождение, следуйте текущему issue.

### 5. Откройте pull request

Когда ваш файл рассылки готов:

1. Закоммитьте изменения в свой форк.
2. Откройте pull request в `ZecHub/zechub` в ветку `main`.
3. Используйте заголовок, соответствующий выпуску, например `Zcash Ecosystem Digest | May 30th`.
4. Укажите issue в теле pull request, чтобы ревьюеры могли связать работу с задачей.

Пример тела pull request:

```md
Closes #ISSUE_NUMBER

Summary:
- Adds the Zcash Ecosystem Digest for Month Day.
- Uses the newsletter template and the current issue deadline.
- Checks links and descriptions for the requested week.
```

После открытия pull request следите за комментариями ревью. Если ZecHub попросит внести правки, обновите ту же ветку вместо открытия второго pull request для того же выпуска.

Сравнивая свою работу с примером, обратите внимание на расположение файла, формат заголовка, порядок разделов, описания ссылок и на то, связан ли pull request с правильной задачей.

### Распространённые ошибки, которых следует избегать

- Открытие pull request до подтверждения даты выпуска или задачи.
- Работа над issue, для которого уже существует связанный pull request.
- Отправка pull request в собственный форк вместо `ZecHub/zechub`.
- Использование неправильного имени файла или размещение файла вне папки `newsletter`.
- Копирование старого выпуска без обновления всех дат, ссылок и описаний.
- Добавление ссылок не за ту неделю.
- Наличие битых ссылок, дублирующихся ссылок или шаблонного текста-заполнителя.
- Открытие нового pull request после комментариев ревью вместо обновления исходной ветки.

### Финальный чек-лист

Прежде чем запрашивать ревью, убедитесь, что:

- Дата issue или задачи совпадает с вашим файлом рассылки.
- Ни один другой открытый pull request уже не покрывает тот же issue или выпуск.
- Файл находится в папке `newsletter`.
- Разделы шаблона заполнены полностью.
- Каждая ссылка работает и имеет полезное описание.
- Тело pull request ссылается на правильный issue.
- Вы готовы внести правки, если ревьюеры попросят изменения.
