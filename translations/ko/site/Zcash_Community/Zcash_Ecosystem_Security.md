# Zcash 생태계 보안

## 생태계 보안 리더

ZCG 투자금을 통해 설립된 Zcash 생태계 보안 리더 역할은 ECC 및 ZF 외부에서 특히 ZCG 수혜자들을 포함한 더 넓은 Zcash 생태계에 전문적인 보안 엔지니어링을 제공하기 위해 만들어졌습니다.

- **2022–2023:** [earthrise](https://forum.zcashcommunity.com/t/zcash-ecosystem-security-lead/42090)가 첫 번째 생태계 보안 리더로 활동했습니다. 더 많은 정보는 [zecsec.com](https://zecsec.com)에서 확인할 수 있습니다.
- **2024–2025:** ZCG는 새로운 [RFP](https://forum.zcashcommunity.com/t/rfp-zcash-ecosystem-security-lead-2023/45723)를 통해 [Least Authority](https://leastauthority.com)를 생태계 보안 리더 역할을 이어가게 선택했습니다. 업데이트 내용은 여기에서 확인할 수 있습니다: [here](https://forum.zcashcommunity.com/t/grant-update-zcash-ecosystem-security-lead/47541).
- **2026:** Shielded Labs는 보안 역량을 강화하기 위해 [Taylor Hornby](https://forum.zcashcommunity.com/t/shielded-labs-engages-taylor-hornby-as-security-consultant/55421)를 보안 컨설턴트로 고용했습니다.

## ZCG 보안 및 취약점 공개 이니셔티브

[ZCG 보안 및 취약점 공개 이니셔티브](https://forum.zcashcommunity.com/t/zcg-security-vulnerability-disclosure-initiative/55545)는 Zcash 생태계 전반에 걸쳐 보안 취약점을 조율된 방식으로 공개하는 프레임워크를 제공합니다.

## 최근 보안 업데이트 (2026)

- **Zebra 4.4.1 (2026년 5월):** [중요한 보안 수정](https://forum.zcashcommunity.com/t/zebra-4-4-1-critical-security-fix/55588)이 출시되었습니다. 모든 노드 운영자는 즉시 업그레이드를 권장합니다.
- **Zebra 4.3.1 (2026년 4월):** [중요한 보안 수정, 도커 기반 채굴 및 CI 강화](https://forum.zcashcommunity.com/t/zebra-4-3-1-critical-security-fixes-dockerized-mining-and-ci-hardening/55389)가 출시되었습니다.
- **여러 취약점 수정 (2026년 4월):** [Zcash의 여러 취약점이 성공적으로 수정](https://forum.zcashcommunity.com/t/several-zcash-vulnerabilities-successfully-remediated/55388)되었습니다. 사용자 자금이나 프라이버시에 영향을 주지 않았습니다.
- **zcashd 공지 (2026년 4월):** [트래픽을 Zebra를 통해 전달하여 zcashd의 공격 표면 축소](https://forum.zcashcommunity.com/t/advisory-reduce-your-zcashd-attack-surface-by-shielding-it-behind-zebra/55390)에 대한 조언이 발표되었습니다.

## 책임 있는 공개

Electric Coin Company와 Zcash Foundation 모두 이 [책임 있는 공개](https://github.com/RD-Crypto-Spec/Responsible-Disclosure/tree/d47a5a3dafa5942c8849a93441745fdd186731e6) 표준을 따르며 다음의 예외가 있습니다:

> "Zcash는 강력한 프라이버시를 제공하는 기술입니다. 노트는 목적지로 암호화되고, 이후 통화 기반은 실제 Zcash 소유자만이 생성할 수 있는 제로 지식 증명을 통해 유지됩니다. 이 과정에서 문제가 발생하고 위조 버그가 생기면, 그 버그를 악용하는 사람이나 해당 버그를 악용한 블록체인 데이터를 식별하는 방법이 없어질 수 있습니다. 따라서 이전 지점의 롤백은 다른 프로젝트에서 실행된 바 있지만 Zcash에서는 불가능합니다. 표준은 취약점을 보고할 때 문제의 전체 세부 사항을 포함하여 재현하도록 요구합니다. 예를 들어 외부 연구자가 실제로 보안 문제가 존재하고 그 문제가 주장하는 영향을 미치는지 증명해야 하는 경우에 이는 필수적입니다. 개발 팀이 문제를 정확히 우선순위를 두고 해결할 수 있도록 하기 위해서입니다. 그러나 위조 버그의 경우, 예를 들어 CVE-2019-7167와 같이, 우리가 그들이 취약하다는 것을 확신하는 한, 조율된 발표 전에 파트너에게 보고서에 세부 사항을 포함하지 않기로 결정할 수도 있습니다."
