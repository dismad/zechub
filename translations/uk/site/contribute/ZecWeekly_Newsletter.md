# Розсилка ZecWeekly

ZecWeekly — це розсилка, яка виходить щоп’ятниці вранці. Вона містить усі новини, що сталися протягом тижня в екосистемі Zcash.

Новини щотижня кураторуються учасниками спільноти, і до розсилки додаються всі релевантні посилання.

Будь ласка, підпишіться на розсилку [тут](https://zechub.substack.com/).

## Долучитися

Найкраще внески до розсилки працюють тоді, коли один дописувач готує випуск для правильного тижня, стежить за поточним bounty або координаційним тредом і надсилає pull request після того, як щотижневі посилання готові. Будь ласка, не надсилайте майбутній випуск до того, як ZecHub опублікує або підтвердить дату цього випуску. Завчасні pull request-и часто пропускають оновлення наприкінці тижня, конфліктують із призначеним куратором або використовують неправильний дедлайн.

### 1. Підтвердьте поточний випуск

Перш ніж почати писати:

- Перевірте [issues ZecHub на GitHub](https://github.com/ZecHub/zechub/issues) і [Dework](https://app.dework.xyz/zechub-2424) щодо поточного завдання розсилки.
- Використовуйте дату в заголовку issue або описі завдання як джерело істини.
- Відкрийте issue і перевірте, чи інший дописувач уже залишив коментар, був призначений або відкрив пов’язаний pull request.
- Перш ніж почати, пошукайте серед відкритих pull request-ів номер issue і дату випуску. Наприклад, пошук `is:pr is:open "May 30th" repo:ZecHub/zechub`.
- Якщо завдання незрозуміле, запитайте в issue, у Discord ZecHub або напишіть [ZecHub у Twitter](https://twitter.com/ZecHub), перш ніж готувати повний випуск.

![Відкриті issues GitHub, відфільтровані за поточними завданнями розсилки ZecWeekly](assets/zecweekly-current-task-search.png)

### 2. Зробіть fork репозиторію

Якщо ви новачок у GitHub, скористайтеся цим процесом:

1. Відкрийте [репозиторій ZecHub](https://github.com/ZecHub/zechub).
2. Натисніть **Fork** і створіть fork у своєму GitHub-акаунті.
3. У своєму fork створіть нову гілку для випуску. Зрозуміла назва гілки буде корисною, наприклад `digest-may-30-2026`.
4. Переконайтеся, що ваш pull request буде спрямований у `ZecHub/zechub` як базовий репозиторій і в `main` як базову гілку.

Якщо ви використовуєте командний рядок, той самий процес виглядає так:

```bash
git clone https://github.com/YOUR-USERNAME/zechub.git
cd zechub
git checkout -b digest-month-day-year
```

### 3. Створіть файл розсилки

Використовуйте [шаблон розсилки](https://github.com/ZecHub/zechub/blob/main/newsletter/newslettertemplate.md) як відправну точку. Випуски розсилки мають розміщуватися в папці [`newsletter`](https://github.com/ZecHub/zechub/tree/main/newsletter).

Під час створення файлу:

- Дотримуйтеся формату імені файлу, запитаного в issue або використаного в нещодавно прийнятих випусках.
- Зберігайте той самий порядок розділів, що й у шаблоні, якщо завдання не вимагає іншого формату.
- Додавайте посилання лише за відповідний тиждень.
- Напишіть короткий і зрозумілий опис до кожного посилання, щоб читачі розуміли, чому воно важливе.
- За потреби перекладайте або стисло викладайте неангломовні джерела англійською.
- Перевірте кожне посилання перед відкриттям pull request.

### 4. Збирайте посилання в правильний час

ZecWeekly зазвичай охоплює активність екосистеми Zcash за поточний тиждень і публікується ближче до кінця тижня. Найбезпечніший таймінг такий:

- Починайте збирати посилання після того, як буде опубліковано поточне issue або завдання для розсилки.
- Ведіть чернетку, поки тиждень ще триває.
- Надсилайте pull request ближче до запитаної дати подання, після того як перевірите оновлення наприкінці тижня.
- Не надсилайте розсилку на майбутній тиждень до того, як з’явиться завдання на цю дату або до того, як ZecHub підтвердить, що саме ви маєте її підготувати.

Якщо в issue сказано подати до конкретної дати, дотримуйтеся саме цієї дати. Якщо між цією сторінкою та поточним issue є суперечність, дотримуйтеся поточного issue.

### 5. Відкрийте pull request

Коли ваш файл розсилки готовий:

1. Закомітьте свої зміни у свій fork.
2. Відкрийте pull request до `ZecHub/zechub` у гілку `main`.
3. Використайте заголовок, який відповідає випуску, наприклад `Zcash Ecosystem Digest | May 30th`.
4. Додайте посилання на issue в тілі pull request, щоб рев’юери могли пов’язати роботу із завданням.

Приклад тіла pull request:

```md
Closes #ISSUE_NUMBER

Summary:
- Adds the Zcash Ecosystem Digest for Month Day.
- Uses the newsletter template and the current issue deadline.
- Checks links and descriptions for the requested week.
```

Після відкриття pull request стежте за коментарями рев’ю. Якщо ZecHub попросить внести правки, оновлюйте ту саму гілку замість відкриття другого pull request для того самого випуску.

### Поширені помилки, яких слід уникати

- Відкриття pull request до того, як підтверджено дату випуску або завдання.
- Робота над issue, яке вже має пов’язаний pull request.
- Надсилання pull request у власний fork замість `ZecHub/zechub`.
- Використання неправильного імені файлу або розміщення файлу поза папкою `newsletter`.
- Копіювання старого випуску без оновлення кожної дати, посилання та опису.
- Додавання посилань не з того тижня.
- Залишення битих посилань, дубльованих посилань або тексту-заповнювача з шаблону.
- Відкриття нового pull request після коментарів рев’ю замість оновлення початкової гілки.

### Фінальний контрольний список

Перш ніж запитувати рев’ю, переконайтеся, що:

- Дата issue або завдання збігається з вашим файлом розсилки.
- Жоден інший відкритий pull request уже не охоплює те саме issue або випуск.
- Файл знаходиться в папці `newsletter`.
- Усі розділи шаблону заповнені.
- Кожне посилання працює й має корисний опис.
- У тілі pull request є посилання на правильне issue.
- Ви доступні для внесення правок, якщо рев’юери попросять зміни.
