# Розсилка ZecWeekly

ZecWeekly — це розсилка, яка виходить щонеділі вранці. Вона включає всі новини, що сталися протягом тижня в екосистемі Zcash. Новини щотижня відбираються учасниками спільноти, і до розсилки додаються всі відповідні посилання. Будь ласка, підпишіться на розсилку [тут](https://zechub.substack.com/).

## Долучитися

Найкраще працює такий формат внеску до розсилки: один учасник готує випуск для відповідного тижня, стежить за поточним тредом bounty або координації та надсилає pull request після того, як щотижневі посилання вже зібрані. Будь ласка, не надсилайте майбутній випуск до того, як ZecHub опублікує або підтвердить дату цього випуску. Передчасні pull request часто пропускають оновлення наприкінці тижня, конфліктують із призначеним куратором або використовують неправильний дедлайн.

### 1. Підтвердьте поточний випуск

Перш ніж почати писати:

- Перевірте [ZEC Bounties ](https://bounties.zechub.wiki/) для поточного завдання розсилки.
- Дочекайтеся призначення

### 2. Зробіть fork репозиторію

Якщо ви новачок у GitHub, скористайтеся цим робочим процесом:

1. Відкрийте [репозиторій ZecHub](https://github.com/ZecHub/zechub).
2. Натисніть **Fork** і створіть fork у своєму GitHub-акаунті.
3. У своєму fork створіть нову гілку для випуску. Зрозуміла назва гілки буде корисною, наприклад `digest-may-30-2026`.
4. Переконайтеся, що ваш pull request буде спрямований до `ZecHub/zechub` як базового репозиторію, а `main` — як базової гілки.

Якщо ви користуєтеся командним рядком, той самий робочий процес виглядає так:

```bash
git clone https://github.com/YOUR-USERNAME/zechub.git
cd zechub
git checkout -b digest-month-day-year
```

### 3. Створіть файл розсилки

Використайте [шаблон розсилки](https://github.com/ZecHub/zechub/blob/main/newsletter/newslettertemplate.md) як відправну точку. Випуски розсилки мають розміщуватися в папці [`newsletter`](https://github.com/ZecHub/zechub/tree/main/newsletter).

Під час створення файлу:

- Дотримуйтеся формату назви файлу, зазначеного в issue або використаного в нещодавно прийнятих випусках.
- Зберігайте той самий порядок розділів, що й у шаблоні, якщо тільки в завданні не вказано інший формат.
- Додавайте посилання лише з відповідного тижня.
- Пишіть короткий і зрозумілий опис для кожного посилання, щоб читачі розуміли, чому воно важливе.
- За потреби перекладайте або стисло викладайте неангломовні джерела англійською.
- Перевірте кожне посилання перед відкриттям pull request.

### 4. Збирайте посилання в правильний час

ZecWeekly зазвичай охоплює активність в екосистемі Zcash за поточний тиждень і публікується ближче до кінця тижня. Найбезпечніший час для цього:

- Починайте збирати посилання після того, як буде опубліковано поточний випуск розсилки або завдання.
- Тримайте чернетку, поки тиждень ще триває.
- Надсилайте pull request ближче до запитаної дати подання, після того як перевірите оновлення наприкінці тижня.
- Не надсилайте розсилку для майбутнього тижня до того, як з’явиться завдання на цю дату або поки ZecHub не підтвердить, що саме вам слід її підготувати.

Якщо в issue сказано подати до конкретної дати, дотримуйтеся цієї дати. Якщо є суперечність між цією сторінкою та поточним issue, дотримуйтеся поточного issue.

### 5. Відкрийте pull request

Коли ваш файл розсилки готовий:

1. Закомітьте свої зміни у свій fork.
2. Відкрийте pull request до `ZecHub/zechub` у гілку `main`.
3. Використайте заголовок, що відповідає випуску, наприклад `Zcash Ecosystem Digest | May 30th`.
4. Додайте посилання на issue в описі pull request, щоб рев’юери могли пов’язати роботу із завданням.

Приклад опису pull request:

```md
Closes #ISSUE_NUMBER

Summary:
- Adds the Zcash Ecosystem Digest for Month Day.
- Uses the newsletter template and the current issue deadline.
- Checks links and descriptions for the requested week.
```

Після відкриття pull request стежте за коментарями рев’ю. Якщо ZecHub попросить внести зміни, оновіть ту саму гілку замість того, щоб відкривати другий pull request для того самого випуску.

### Типові помилки, яких слід уникати

- Відкриття pull request до того, як дату випуску або завдання буде підтверджено.
- Робота над issue, до якого вже прив’язаний pull request.
- Надсилання pull request до власного fork замість `ZecHub/zechub`.
- Використання неправильної назви файлу або розміщення файлу поза папкою `newsletter`.
- Копіювання старого випуску без оновлення кожної дати, посилання й опису.
- Додавання посилань не з того тижня.
- Залишення неробочих посилань, дублікатів посилань або тексту-заповнювача з шаблону.
- Відкриття нового pull request після коментарів рев’ю замість оновлення початкової гілки.

### Підсумковий контрольний список

Перш ніж запитувати рев’ю, переконайтеся, що:

- Дата в issue або завданні відповідає вашому файлу розсилки.
- Жоден інший відкритий pull request уже не охоплює те саме issue або випуск.
- Файл розміщено в папці `newsletter`.
- Усі розділи шаблону заповнені.
- Кожне посилання працює та має корисний опис.
- В описі pull request є посилання на правильне issue.
- Ви готові внести правки, якщо рев’юери попросять змін.
