# Розсилка ZecWeekly

ZecWeekly — це розсилка, яка виходить щоп’ятниці вранці. Вона містить усі новини, що сталися протягом тижня в екосистемі Zcash.

Новини щотижня відбираються учасниками спільноти, а всі відповідні посилання додаються до розсилки.

Будь ласка, підпишіться на розсилку [тут](https://zechub.substack.com/).

## Долучитися

Найкраще внески до розсилки працюють тоді, коли один дописувач готує випуск для правильного тижня, стежить за поточною гілкою баунті або координації та надсилає pull request після того, як щотижневі посилання готові. Будь ласка, не надсилайте майбутній випуск до того, як ZecHub опублікує або підтвердить дату цього випуску. Передчасні pull request часто пропускають оновлення наприкінці тижня, конфліктують із призначеним куратором або використовують неправильний дедлайн.

### 1. Підтвердьте поточний випуск

Перш ніж почати писати:

- Перевірте [issues ZecHub на GitHub](https://github.com/ZecHub/zechub/issues) і [Dework](https://app.dework.xyz/zechub-2424) на наявність поточного завдання для розсилки.
- Використовуйте дату в заголовку issue або описі завдання як головне джерело істини.
- Відкрийте issue і перевірте, чи інший дописувач уже не залишив коментар, не був призначений або не відкрив пов’язаний pull request.
- Перш ніж починати, пошукайте відкриті pull request за номером issue і датою випуску. Наприклад, пошук `is:pr is:open "May 30th" repo:ZecHub/zechub`.
- Якщо завдання незрозуміле, запитайте в issue, у Discord ZecHub або напишіть [ZecHub у Twitter](https://twitter.com/ZecHub), перш ніж готувати повний випуск.

![Відкриті issues на GitHub, відфільтровані за поточними завданнями розсилки ZecWeekly](assets/zecweekly-current-task-search.png)

### 2. Зробіть форк репозиторію

Якщо ви новачок на GitHub, скористайтеся цим робочим процесом:

1. Відкрийте [репозиторій ZecHub](https://github.com/ZecHub/zechub).
2. Натисніть **Fork** і створіть форк у своєму обліковому записі GitHub.
3. У своєму форку створіть нову гілку для випуску. Зрозуміла назва гілки буде корисною, наприклад `digest-may-30-2026`.
4. Переконайтеся, що ваш pull request буде спрямовано до `ZecHub/zechub` як базового репозиторію, а `main` буде базовою гілкою.

Якщо ви користуєтеся командним рядком, той самий робочий процес виглядає так:

```bash
git clone https://github.com/YOUR-USERNAME/zechub.git
cd zechub
git checkout -b digest-month-day-year
```

### 3. Створіть файл розсилки

Використайте [шаблон розсилки](https://github.com/ZecHub/zechub/blob/main/newsletter/newslettertemplate.md) як вихідну точку. Випуски розсилки мають розміщуватися в папці [`newsletter`](https://github.com/ZecHub/zechub/tree/main/newsletter).

Під час створення файлу:

- Дотримуйтеся формату назви файлу, який вимагається в issue або використовується в нещодавно прийнятих випусках.
- Зберігайте той самий порядок розділів, що й у шаблоні, якщо тільки завдання не вимагає іншого формату.
- Додавайте посилання лише за відповідний тиждень.
- Напишіть короткий і зрозумілий опис для кожного посилання, щоб читачі розуміли, чому воно важливе.
- За потреби перекладайте або стисло викладайте неангломовні джерела англійською.
- Перевірте кожне посилання перед відкриттям pull request.

### 4. Збирайте посилання в правильний час

ZecWeekly зазвичай охоплює активність екосистеми Zcash за поточний тиждень і публікується ближче до кінця тижня. Найбезпечніший час такий:

- Починайте збирати посилання після того, як буде опубліковано поточне issue або завдання для розсилки.
- Тримайте чернетку, поки тиждень ще триває.
- Надсилайте pull request ближче до запитаної дати подання, після того як перевірите оновлення наприкінці тижня.
- Не надсилайте розсилку за майбутній тиждень до того, як з’явиться завдання на цю дату або поки ZecHub не підтвердить, що саме вам слід її підготувати.

Якщо в issue сказано подати до певної дати, дотримуйтеся цієї дати. Якщо між цією сторінкою та поточним issue є суперечність, дотримуйтеся поточного issue.

### 5. Відкрийте pull request

Коли ваш файл розсилки готовий:

1. Закомітьте свої зміни у свій форк.
2. Відкрийте pull request до `ZecHub/zechub` у гілку `main`.
3. Використайте заголовок, що відповідає випуску, наприклад `Zcash Ecosystem Digest | May 30th`.
4. Додайте посилання на issue в тексті pull request, щоб рев’юери могли пов’язати роботу із завданням.

Приклад тексту pull request:

```md
Closes #ISSUE_NUMBER

Summary:
- Adds the Zcash Ecosystem Digest for Month Day.
- Uses the newsletter template and the current issue deadline.
- Checks links and descriptions for the requested week.
```

Після того як pull request відкрито, стежте за коментарями до рев’ю. Якщо ZecHub попросить внести правки, оновіть ту саму гілку замість того, щоб відкривати другий pull request для того самого випуску.

### Типові помилки, яких слід уникати

- Відкриття pull request до того, як дату випуску або завдання буде підтверджено.
- Робота над issue, до якого вже прив’язаний pull request.
- Надсилання pull request до власного форку замість `ZecHub/zechub`.
- Використання неправильної назви файлу або розміщення файлу поза папкою `newsletter`.
- Копіювання старого випуску без оновлення кожної дати, посилання та опису.
- Додавання посилань не з того тижня.
- Залишення неробочих посилань, дубльованих посилань або тексту-заповнювача із шаблону.
- Відкриття нового pull request після коментарів рев’ю замість оновлення початкової гілки.

### Підсумковий контрольний список

Перш ніж запитувати рев’ю, переконайтеся, що:

- Дата issue або завдання збігається з вашим файлом розсилки.
- Жоден інший відкритий pull request уже не охоплює той самий issue або випуск.
- Файл розміщено в папці `newsletter`.
- Усі розділи шаблону заповнені.
- Кожне посилання працює та має корисний опис.
- Текст pull request містить посилання на правильний issue.
- Ви доступні для внесення правок, якщо рев’юери попросять про зміни.
